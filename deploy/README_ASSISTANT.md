# AI-ассистент paper-desk (Telegram + терминал)

Спросить бота, что происходит, с телефона или из терминала. Отвечает по-русски,
разбирает состояние обоих боевых контуров, логи и systemd.

**Сейчас ассистент работает только на чтение.** Он физически не может ни
остановить торговлю, ни изменить файл: инструментов записи в коде нет. Kill-switch
с подтверждением — вторая фаза (см. конец файла).

---

## 1. Что это и почему так устроено

- **python3 + только stdlib.** Ни pip, ни venv: код доезжает на VPS обычным
  `git pull` внутри торгового тика и сразу работоспособен. Язык намеренно
  отличается от pwsh движков — ассистент не может попасть в критический путь тика.
- **Long-polling, не webhook.** Не нужен ни белый IP, ни TLS-сертификат, только
  исходящие соединения.
- **Состояние вне git-дерева** (`/var/lib/trading-assistant`): тик каждую минуту
  делает `git pull --rebase --autostash`, лишние файлы в дереве дали бы конфликты.

Схема:

```
Telegram ──getUpdates──> assistant/bot.py ──┐
                                            ├─> agent.py ──> OpenRouter (DeepSeek/Gemini)
ноут: tools/ask.ps1 ──ssh──> assistant/cli.py ┘        └──> tools_impl.py (только чтение)
```

## 2. Установка на VPS

### 2.1 Ключ OpenRouter

Отдельный env-файл, чтобы CLI по ssh работал без доступа к брокерским ключам:

```bash
sudo tee /etc/trading-assistant.env >/dev/null <<'EOF'
OPENROUTER_API_KEY=sk-or-v1-...
ASSISTANT_MODEL=deepseek/deepseek-chat
ASSISTANT_MODEL_FALLBACK=google/gemini-2.0-flash-001
EOF
sudo chown root:trader /etc/trading-assistant.env
sudo chmod 640 /etc/trading-assistant.env
```

`640 root:trader` — принцип наименьших привилегий: ассистенту нужен только ключ
модели, а `/etc/trading-live.env` (0600 root:root, с ключами Bybit и T-Invest)
читает лишь systemd от root.

### 2.2 Кто может писать боту

По умолчанию — `TG_CHAT_ID` из `/etc/trading-live.env`. Явно задать список:

```bash
echo 'ASSISTANT_TG_ALLOWED_CHATS=123456789' | sudo tee -a /etc/trading-assistant.env
```

`TG_CHAT_ID_FUT` сюда **не добавлять**: это второй получатель алертов, а не
оператор бота. Сообщения от всех остальных игнорируются молча.

### 2.3 Проверка до запуска демона

Сначала CLI — без Telegram, без денег, без риска:

```bash
cd /home/trader/paper-desk
python3 -m assistant.cli --snapshot          # состояние, без модели вообще
set -a; . /etc/trading-assistant.env; set +a
echo "что с ботом" | python3 -m assistant.cli --stdin
```

Прогнать офлайн-тесты (сеть и ключ не нужны):

```bash
python3 assistant/tests/run_all.py
```

### 2.4 Демон

**Перед первым запуском убедись, что этот токен больше никто не поллит** —
два `getUpdates` на одном токене несовместимы (Telegram отдаёт 409). В частности,
проверь, что выключен старый бот openclaw.

```bash
sudo cp deploy/trading-assistant.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now trading-assistant
journalctl -u trading-assistant -f
```

Положительный тест «поллер один»: `curl "https://api.telegram.org/bot<TOKEN>/getUpdates"`
теперь должен вернуть **409 Conflict**. Это ожидаемо и правильно.

## 3. Терминал на ноуте

```powershell
$env:ASSISTANT_VPS = "trader@1.2.3.4"     # или tools\ask.config.json (в .gitignore)

.\tools\ask.ps1 "что с ботом"
.\tools\ask.ps1 "почему не было входов сегодня"
.\tools\ask.ps1 -Interactive              # REPL
.\tools\ask.ps1 -Snapshot                 # состояние без модели, бесплатно
.\tools\ask.ps1 -Logs rf -Lines 80        # сырой хвост лога, без модели
```

Логика и ключи живут только на VPS. Заодно это единственный способ увидеть с ноута
живые `tick_log.txt` и `journalctl` — в git они не попадают никогда (`.gitignore`).

## 5. Инструменты, доступные модели

Все — только чтение. Каждый результат обрезается до 6000 символов и проходит через
скраббер секретов.

`get_state` · `get_signals` · `get_open_positions` · `tail_log` · `journalctl_tail`
· `systemctl_status` · `host_health` · `list_halt_files` · `git_status` · `git_log`
· `read_repo_file` · `search_journals`

`get_signals` — главный ответ на «почему не было входов»: возвращает
`сводка_блокирующих_ворот`, то есть какие именно ворота стратегии срубили сделки.

## 6. Безопасность

- **Инъекции через логи.** Содержимое лога может содержать текст, похожий на приказ
  модели. Защита не в уговорах промпта, а в полномочиях: модель не может ничего
  исполнить, инструментов записи не существует. Плюс `untrusted_content: true` в
  каждом конверте и явное правило в системном промпте.
- **Секреты.** Инструмента чтения переменных окружения нет. Скраббер вычищает
  фактические значения `TG_BOT_TOKEN`, `OPENROUTER_API_KEY`, `BYBIT_*`, `TINVEST_*`
  из вывода каждого инструмента **и** из финального ответа. `read_repo_file` закрыт
  для `.secrets/`, `vps/`, `.git/`, `*.env`, `*.enc`.
- **Лимиты.** 20 сообщений / 10 минут на чат; дневной потолок токенов
  (`ASSISTANT_DAILY_TOKENS`, по умолчанию 300k) и вызовов; не более 6 раундов
  инструментов за ход; входящее сообщение длиннее 2000 символов отклоняется.
- **Ресурсы.** `MemoryMax=300M`, `CPUQuota=40%` — у торговых тиков бюджет 110 секунд,
  ассистент не должен с ними конкурировать.

## 7. Диагностика

| Симптом | Причина и что делать |
|---|---|
| В логе `КОНФЛИКТ: 409` | Токен поллит кто-то ещё. Найти и выключить второй экземпляр или старого бота. |
| `OPENROUTER_API_KEY не задан` | `/etc/trading-assistant.env` не читается. Проверить права `640 root:trader`. |
| Бот молчит на сообщение | chat_id не в whitelist — так и задумано. Свой id: `journalctl -u trading-assistant \| grep игнор`. |
| Ответ есть, но данные старые | Смотри возраст тика в `/статус`. Если он большой — проблема в контуре, а не в ассистенте. |
| `tail_log`: файла нет | Логи существуют только на VPS. С ноута их не видно принципиально. |

Ассистент никогда не должен влиять на торговлю. Если есть хоть тень сомнения:
`sudo systemctl stop trading-assistant` — торговые тики от этого не зависят вообще.

## 8. Действия: ручное закрытие paper-сделок (сделано 2026-07-20)

Первое и пока единственное действие ассистента — закрыть фьючерсную позицию
БУМАЖНОГО контура (C2/C3b, зеркально в обоих профилях). Живые контуры
(T-Invest, Bybit) ассистент по-прежнему трогать не умеет.

### Как пользоваться

- Свободным текстом: «закрой позицию по нефти», «зафиксируй прибыль по ртс» —
  модель вызывает `propose_close_position`, приходят кнопки Подтвердить/Отмена.
- Или командой `/позиции` (без модели, бесплатно): список открытых позиций
  с кнопкой «Закрыть» у каждой.
- После нажатия «Подтвердить» ассистент пишет заявку в
  `data/rf/manual_close_req.json` и сразу же сам коммитит и пушит её
  (`actions.push_request_now`, под тем же flock `/run/lock/live-rf-tick.lock`,
  что и RF-тик; при любой неудаче молча отступает — fast-path в
  `live_rf_tick.sh` допушит в течение минуты). Push триггерит **лёгкий ран**
  `manual-close.yml` (только `rf_engine.ps1` + коммит, ~1 мин; `tick.yml`
  этот файл игнорирует через paths-ignore), движок закрывает по последней
  часовой цене ISS с реальным слипом (`Invoke-RfManualClose`), результат
  приезжает обратно git pull'ом, и бот присылает отчёт с ценой, P&L и R.
  Итоговая задержка обычно **1–3 мин**; worst case ~15 мин, если
  подтверждение попало на середину планового полного тика (concurrency-
  очередь). Дашборд подхватит закрытие ближайшим плановым тиком.
- В `rf_trades.json` такие сделки помечены `exitReason='manual-tg'` — при
  сравнении форвард-теста с бэктестом их можно исключать. Re-arm после ручного
  закрытия работает как после обычного стопа (быстрый перезаход по укороченному
  каналу, если тренд подтвердит себя).

### Безопасность

Модель может только ПРЕДЛОЖИТЬ: токен подтверждения генерит код
(`secrets.token_hex`), в вывод модели и в текст чата он не попадает — живёт
только в `pending.json` и в callback_data кнопок. Подтвердить может только
нажатие кнопки из whitelist-чата (проверяются и чат, и нажавший). TTL
подтверждения 180 с; повторное нажатие и дубль заявки на тот же актив —
no-op (трёхуровневый дедуп + реестр обработанных `req_id` в res-файле движка).
Каждая створка пишется в `actions_audit.log`.

### Обкатка и снятие DRY

Под `ASSISTANT_DRY_ACTIONS=1` (по умолчанию) подтверждение пишет заявку в
песочницу `$STATE_DIR/sandbox/` и помечает ответ `[DRY]` — полный прогон UX
без эффекта. Снятие: `ASSISTANT_DRY_ACTIONS=0` в `/etc/trading-assistant.env`,
`sudo systemctl restart trading-assistant`, затем боевой тест на живой
paper-позиции. Диагностика: `grep manual-close` в `journalctl -u
trading-assistant`; заявка видна коммитом «manual-close request» в main.
